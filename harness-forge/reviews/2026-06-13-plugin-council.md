## Converged findings → 0.3.1 dispositions

- **CV1 — The vendored `_lattice.py` copy drifts with no gate** (Charity M. *Critical*, David F. *Major*, Simon W.). `wire.py apply` byte-copies `bin/lattice.py` → the user's `.harness/hooks/_lattice.py`; on a plugin update nothing re-applies, `wire.py check` only WARNs (returns exit 0), and CI can't see it (apply+check run microseconds apart on identical bytes). The staleness plugin shipped a staleness bug. → **Fixed:** `KERNEL_VERSION` stamped in `lattice.py`, recorded beside the copy at apply, and `wire.py check` now **exits non-zero** on version-or-byte drift (the H3 anchor stops green-lighting a stale kernel); a CI step proves apply → mutate-kernel → check-fails → re-apply → check-clean. `lattice.save()` is now **atomic** (temp + `os.replace`) so the on-every-edit cascade can't truncate canonical state.
- **CV2 — `harness-builder` is unreachable dead weight** (Elon M. *Critical*, Boris M., Steve Y.). No command dispatches it; it duplicates the operating-procedure loop the 5 commands already drive; the ROADMAP admits its `/harness-run` auto-loop is unbuilt — *the identical defect the 0.1.1 council deleted `cell-regenerator` for* (`reviews/2026-06-12...:16` vs `:22`), kept only on "it was requested." → **Fixed:** **deleted.** It returns in the same release as `/harness-run` + the wired stop-gate, when it finally has a reachable loop. Net roster: 11 agents.

# Changelog

## 0.5.12 — 2026-06-15

- **`validate.py` reports the ACTUAL resulting maturity** (a dev-factory dogfooding finding, DF-6 — benefits harness-forge identically). On a passing verifier the function advanced the cell conditionally (`defined→instantiated→validated`, gated by `transition_ok`) but returned a **hardcoded** `"PASS — {cell} → validated"` *regardless of whether the transition fired*. Re-validating a `stale` cell therefore printed `→ validated` and exited 0 while the cell stayed `stale` (the FSM routes `stale → regenerating → validated`, so `transition_ok(stale, "validated")` is False) — a caller trusting stdout believed a stale cell was cleared. Now it reports `before → after` and, when the advance to `validated` didn't fire, says so and names the `regenerating` route. Message-only — the maturity logic is unchanged, **`KERNEL_VERSION` stays `0.5.2`** (an already-wired project reads STALE on the byte-drift; `wire.py apply` re-syncs, the normal flow). plugin.json 0.5.11 → 0.5.12.

## 0.5.0 — 2026-06-13

- **v0.5.0 — the arming gap is closed (I-9): the loop-active marker.** The one genuine residual all four red-teams kept naming: `/harness-run`'s global caps are *enforced* in code by `gate-budget`, but the budget was *armed* by the orchestrator's step 0 — skip it and the loop was silently unbounded, because the gate couldn't distinguish a loop-write from a human's edit. **The fix is a loop-active marker** (`.harness/run/loop-active.json`), written by `/harness-run` as its mechanical **step 0a** (`run-budget.py mark`) *before* the budget is armed at **step 0b** (`run-budget.py start`). `gate-budget` now denies **every write while the loop is marked active but no budget is armed** — so forgetting step 0b **fails closed**, not unbounded.

## Deferred
<!-- Capabilities considered but postponed ‚Äî document the reason and re-evaluation trigger. -->
- **Live `claude plugin validate` invocation inside the gate** ‚Äî deferred because the CLI may not be present in every CI/headless environment and shelling out couples the gate to a binary version; `validate_plugin.py` reproduces the documented static checks in-repo. Revisit when the library has a guaranteed CLI on its CI runner.
- **Executing a candidate plugin to measure real context cost / behavior** ‚Äî deferred because installing and running third-party plugin code is the exact trust-boundary risk P9 warns about; v0.1 is static-analysis only. Revisit with a sandboxed harness.
- **Auto-generating the whole carved marketplace for this library** ‚Äî deferred until the carve method is validated on ‚â•1 plugin end-to-end; carve currently emits a *proposal*, not an applied marketplace. Revisit after the first real plugin ships.
- **Trimming the v0.1 rubric library** ‚Äî the 2026-06-02 red-team (Elon lens) flagged 5 deep-dive rubrics + 5 foundations authored at N=0 as possible premature elaboration. Kept for v0.1 because the `build-against-the-standard` bridge needs build-side/score-side symmetry (a half-populated bridge is itself a defect, and the user asked for the full foundational set). Re-evaluate after the first ‚â•3 real-plugin applications ‚Äî if the spine proves sufficient, demote under-used drill-downs to the same "author when the spine proves coarse" discipline already applied to P1/P7/P8/P9.
- **~~Co-locating the P1/P7/P8/P9 cross-skill rubric references~~ ‚Äî RESOLVED 2026-06-02 (decision A).** When plugins-studio was packaged as a plugin, the four cross-skill rubrics (`cold-start-orientation`, `skills-authoring`, `skill-extensibility`, `security-and-scope-containment`) were **co-located** into `references/rubrics/` (vendored, zero `../` paths) ‚Äî Simon's predicted P4 self-violation, fixed at packaging time. **Follow-up:** the vendored copies have **no automatic drift check** against the `skills-studio` originals (re-sync is manual) ‚Äî mechanize a checksum/version drift gate in a future version.

## Out of scope (by design)
<!-- Capabilities explicitly excluded ‚Äî not postponed, intentionally outside this skill's boundary. -->
- **Authoring or scoring a single skill's *content*** ‚Äî excluded because that is `skills-studio`'s job; `plugins-studio` operates on the bundle, not the skill body. It consumes skills as components; it does not write them.
- **Renaming / merging / retiring / splitting skills** ‚Äî excluded because that is `skills-refactor`; carve decides *boundaries*, it does not execute library-level skill restructuring.
- **Designing a single MCP server's tool schemas / perimeter** ‚Äî excluded because that is `core-mcp-best-practices`; plugins-studio decides *whether* a capability should be an MCP and that a bundled MCP is well-formed, not the internal tool design.
- **Non-Claude-Code plugin/extension formats** (VS Code extensions, Obsidian plugins, npm packages) ‚Äî excluded; this skill targets the Claude Code plugin + marketplace format specifically. Their *design wisdom* informs the rubrics; their *packaging* does not.
